# 教育版桌面端安全基线与防破解方案

本文档用于沉淀当前项目的防破解、防逆向工程优化建议，作为教育版桌面端（Tauri）上线前的实施清单。

## 当前风险结论（基于现状）

- 当前形态是 Web 主体 + 部分 Tauri 适配，核心安全能力还未完全落地。
- 客户端存在可被逆向后绕过的授权路径，前端仍承担了过多信任边界。
- 本地敏感信息存储策略存在弱点（硬编码密钥、加密逻辑未真正启用）。
- 仓库缺少完整 `src-tauri` 关键文件，安全配置无法被完整审计与版本管理。
- 存在用户代码执行能力（JS/Python 节点），若不分级隔离将放大攻击面。

---

## P0（必须先做，建议 1-2 周）

### 1. 补全并纳管 Tauri 工程

将以下文件纳入仓库并持续维护：

- `src-tauri/Cargo.toml`
- `src-tauri/tauri.conf.json`
- `src-tauri/capabilities/default.json`
- `src-tauri/src/main.rs`（或与当前结构对应的入口）

目标：让能力边界、更新签名、窗口安全策略可审计、可追溯、可复现。

### 2. Tauri 权限最小化（Capabilities）

采用默认拒绝策略，只放行必要权限：

- 允许：目录选择、受限文件写入、路径拼接、受限本地存储
- 禁止：任意 shell 命令、任意路径读写、无约束插件能力

要求：所有 `invoke` 命令做参数校验（路径白名单、长度限制、格式校验）。

### 3. 授权判定后移到服务端

核心原则：客户端仅展示状态，服务端决定是否允许关键操作。

- 新增签名许可证（JWS/Ed25519）：`user_id`、`device_hash`、`plan`、`exp`、`features`、`nonce`
- 客户端只做验签与到期检查
- 关键能力（运行、导出、云写入）仍需服务端二次鉴权

### 4. 设备绑定 + 可撤销许可证

- 首次激活绑定设备指纹（建议多维哈希，不直接暴露原始硬件标识）
- 许可证可吊销、可换绑（有限次数），避免误伤正常用户
- 离线模式设置明确宽限期，超期需回连验证

### 5. 本地敏感存储改造

敏感数据（token/license/refresh-token）不再使用前端硬编码密钥方案。

- Windows：优先使用 DPAPI / Credential Manager
- 通过 Rust 命令暴露 `secure_set/secure_get/secure_delete`
- 前端仅存非敏感配置，敏感信息从 JS 可读层剥离

### 6. 发布版调试面收敛

- 关闭/限制 DevTools
- 移除生产版调试入口和高风险日志
- 保持生产 sourcemap 关闭
- 混淆仅作为“提高逆向成本”，不作为安全边界

## P1（增强项，建议 2-4 周）

### 7. 更新链路防篡改

- 启用 Tauri Updater 签名校验
- 客户端仅接受合法签名更新包
- 拒绝降级安装（anti-rollback）
- 增加灰度策略与异常回滚机制

### 8. Rust 构建硬化

在 release 配置中启用：

- `lto = true`
- `codegen-units = 1`
- `strip = "symbols"`
- `panic = "abort"`

说明：该项主要提升逆向成本与二进制精简，不替代授权安全。

### 9. 高风险能力分级（教育版）

对“用户代码执行”能力进行许可分级：

- 默认关闭 JS/Python 执行节点（教育版基础包）
- 通过许可证 `features` 精准放开
- 打开时给出明确风险提示（仅可信工作流）

### 10. 完整性与风控补强

- 增加关键二进制完整性校验（仅告警，不直接误封）
- 服务端风控：异常设备并发、短时高频调用、重放特征识别
- 行为审计可追踪，支持盗版传播溯源

## 建议改造路线（按文件维度）

### 前端

- `src/common/Storage.tsx`
  - 去除硬编码密钥依赖
  - 敏感字段改走 Rust 安全存储命令
- `src/components/ComActivation.tsx`
  - 激活成功后写入 `license_token`（非仅更新展示态）
- `src/common/service/server.tsx`
  - 关键接口增加许可证与设备证明头
  - 保证所有关键操作都可被服务端拒绝

### Rust / Tauri

- 新增命令：
  - `validate_license`
  - `secure_set`
  - `secure_get`
  - `secure_delete`
  - `save_file_to_path`（加强路径与类型校验）
- 能力配置：
  - capabilities 白名单化
  - CSP 与外链策略收敛

---

## 上线前验收清单（必须通过）

- 篡改本地 token/license 后，关键功能不可越权
- 离线超过宽限期后，进入受控降级而非无限可用
- 复制安装目录到新设备，授权不应直接迁移成功
- 重放激活/运行请求无效（nonce/timestamp 生效）
- 非签名包与降级包更新均被拒绝
- 前端补丁绕过 UI 后，服务端仍可阻断关键能力

---

## 说明

- 防破解是“提高成本 + 缩短盗版收益窗口”的系统工程，不存在绝对不可破解。
- 真正有效的策略是：关键权益上收服务端、客户端最小权限、更新链路强签名、持续风控监测。
